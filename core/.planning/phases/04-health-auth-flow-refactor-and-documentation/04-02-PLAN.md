    Create a practical technical reference at `../docs/system-design/integrations/Nango Sync Script Guide.md`. This is a FULL DOC EXCEPTION per user decision — write complete content, not just a stub.

    The document should be a quick reference (1-2 pages) covering:

    **1. Sync Config Structure**
    - How to define a sync in `nango.yaml` — model name, runs schedule, auto_start, track_deletes
    - Model naming convention: model name must match the `key` field on `CatalogEntityTypeEntity` (this is how the Temporal workflow resolves the entity type)
    - Provider config key usage

    **2. Model and Record Format**
    - Expected record shape: flat key-value payload with an `id` field (or custom `_externalIdField`)
    - How `_externalIdField` in CatalogFieldMapping controls which payload field is the dedup key
    - Relationship fields: lists of external IDs keyed by relationship definition name

    **3. batchSave and Checkpointing**
    - Use `nango.batchSave(records, model)` to persist records in batches
    - Call `await nango.batchSave()` regularly (every 100-500 records) to create checkpoints
    - On retry, Nango resumes from the last checkpoint — records before the checkpoint are not re-sent
    - `batchDelete(records, model)` for soft-delete propagation

    **4. Relationship ID Patterns**
    - External IDs (not internal UUIDs) for relationship targets
    - The sync pipeline resolves external IDs to internal entities in Pass 2
    - Example: a Contact record with `companies: ["ext-company-1", "ext-company-2"]`

    **5. Metadata and Actions**
    - Nango adds `_nango_metadata` with `lastAction` (ADDED/UPDATED/DELETED), `firstSeenAt`, `lastModifiedAt`, `cursor`
    - The pipeline uses `lastAction` to determine upsert vs delete semantics
    - `cursor` is used for incremental sync (modifiedAfter parameter)

    Keep the tone practical and example-driven. Use code blocks for nango.yaml snippets and TypeScript sync script examples. Reference the Riven pipeline behavior (dedup, relationship resolution, error isolation) so script authors understand downstream effects.
  </action>
