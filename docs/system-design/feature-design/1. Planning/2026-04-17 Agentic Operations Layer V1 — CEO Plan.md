## Vision

### Problem Framing (unchanged, validated)

DTC ecom operators at $1M–$20M run 5–10 tools. Every tool generates signals; nobody orchestrates them. Shopify Sidekick Pulse (Winter '26) detects anomalies inside Shopify-native data and fires single-step actions. Riven does what Sidekick can't: detect across the full stack → reason against business definitions → research externally → orchestrate multi-step responses with HITL → measure lift.

"Sidekick detects. Riven orchestrates."

### Strategic Shift in This Review

The original design framed V1 as "UC1: Feedback Intelligence" with orchestration deferred to V2. Review surfaced a thesis-to-product gap: V1 alert-only reads as Sauce AI competitor until V2 lands 16 weeks later.

**Reframe:** V1 = UC1 riding on an agent runtime + NL wizard + scope manifest. The runtime is the foundational primitive. UC1 is the first template running on it. UC2/UC3 become later templates on the same runtime, not new codebases.

Wizard pattern modeled on Claude skills technique: user specifies outcome + acceptable operations scope → agent performs entity ecosystem research, web research, strategic documentation generation, and scoped execution via integration bindings. Declarative capability manifest, allowed-tools scope, structured tool-use loop.

## Accepted Scope (V1)

### Foundational primitive
- **Agent runtime:** NL outcome intake → workspace entity research (reads merchant's workspace entities: products, customers, campaigns, business definitions, recent connected-tool data within a **read-cost budget of ~50K tokens / 15 seconds per full wizard invocation end-to-end (not per tool-use step)** — entities beyond budget are summarized by prior job, not re-read) → web research (V1 scoped to safe sources only: public benchmarks, industry reports, vendor docs; no Ad Library or competitor scraping — deferred to V2 with paid provider evaluation) → strategic documentation generation (analysis doc + plan + recommendations + expected lift) → scoped execution via integration bindings → approval gates → minimal measurement primitive.

**Measurement primitive per seeded template (V1 metrics, specific):**
| Template | Primary metric | Window | Baseline source |
|----------|----------------|--------|-----------------|
| Packaging Complaint Pattern | Packaging-tagged ticket volume/week | 30-day before/after | Rolling 90-day pre-action median |
| VIP Churn Signal | VIP retention rate (active subscriber count) | 60-day before/after | Prior 60-day matched cohort |

Measurement flags confounders when detected (concurrent campaign, seasonality marker, SKU change). Does not claim causal attribution; labels outputs as "before/after comparison" explicitly.

Stubbed templates (Review Sentiment Drop, Discount Cohort Drift, Support Volume Anomaly) inherit measurement design as part of agency co-design — they do not ship without a metric + window + baseline spec. Discount Cohort Drift has the highest measurement stakes (cross-tool attribution) and ships with a default metric spec: "60-day repeat purchase rate of a cohort vs. matched-size non-discount cohort; confounders flagged if concurrent campaigns overlap."
- **Scope manifest:** declarative "allowed operations" config per template/workflow. Format TBD by eng-review; reference implementation pattern is the Claude Code skill manifest (YAML frontmatter + `allowed-tools:` list + scope description). Illustrative stub:
  ```yaml
  template: packaging-complaint-pattern
  allowed_tools:
    - gorgias.tag_ticket
    - klaviyo.draft_flow   # draft only, never send
    - slack.post_approval_card
  guardrails:
    require_approval_for: [klaviyo.*, gorgias.tag_ticket]
    max_actions_per_run: 10
  ```
  **Eng decision required (flagged for /plan-eng-review):** enforcement semantics. Options: hard-fail on out-of-scope tool call, prompt-then-block, or log+continue. CEO plan recommends hard-fail for V1 (trust-first); eng-review decides final.
- **Business Definition Layer primitive:** ties to 2026-04-01 Business Definition Layer plan. **Dependency status:** that plan is ACTIVE (accepted scope, not yet built). V1 of this plan assumes the BDL core (workspace-scoped definitions table + onboarding prompts + query-time resolution + transparency) ships alongside or as part of this V1. **Fallback behavior if BDL slips:** V1 ships a minimal inline defs store (JSON blob in workspace settings, 5-key schema: vip_customer, churn_risk, creative_fatigue, healthy_cohort, alert_routing). Pricing tier "5 business definitions" delivers against inline store; migrates to full BDL in V1.1 when that plan ships. Users not told about the substrate change — UI surface identical.
- **Agent quality gates + fallbacks:** agent outputs run through a minimum-quality check before user sees them (output non-empty, action brief cites at least one definition and one data point, no contradictions against scope manifest). On low-quality output: surface "Riven needs more context" prompt rather than ship a bad brief. Kill switch: workspace-level toggle disables all agent orchestration within 60 seconds.
- **Quality-gate eval set:** labeled corpus of 30+ intentionally-bad + 30+ good agent outputs. **The "bad" half is authored by a signed agency partner** (1–2 hrs of their time, post-MOU) based on what they'd reject as operators — not by Jared. Prevents "self-graded ritual" failure mode flagged by outside voice. Jared authors the "good" half. Threshold targets (95% trip on bad, <5% false-positive on good) are eng-review deliverables validated against this set. If no agency signs by Week -1, fall back to AI-authored bad corpus via adversarial subagent (second-best, still independent from builder).
- **PII + tenancy posture:** Gorgias tickets and customer data contain PII. V1 stance: per-workspace row-level isolation; no cross-tenant model training; LLM calls are transient (no retention beyond execution log). Stated to prospects upfront. EU region deployment not promised in this plan (revisit post-V1 based on EU prospect demand).

## Effort and Timeline Honesty

**Original design:** V1 in 8 weeks for solo founder. Reviewer Concerns (internal) already called this aggressive.

**This plan's V1:** expands to agent runtime + scope manifest + wizard + sandbox + 2 seeded + 3 stubbed templates + landing page + parallel agency partner motion.

**Single canonical timeline:**
- Solo human (no AI leverage): 12–16 weeks
- Solo founder with heavy CC+gstack leverage (this is Jared's actual situation): **10–12 weeks realistic, 8 weeks stretch** (accounts for Shopify OAuth wall-time + agency outbound on top of build). 6 weeks is fantasy-best-case only if every third-party gate short-circuits and agency partners absorb all 3 stubbed-template design work.
- Pre-V1 work (weeks -4 to 0): agency partner outbound (6–10 hrs/wk), landing page (1–2 days with design-consultation skill), assignment tasks from design doc
- GTM + build run in parallel, not serialized

**Critical path (build order):** agent runtime → scope manifest → Gorgias+Shopify integrations → 2 seeded templates → Slack binding → minimal measurement → sandbox. Everything else (3 stubbed templates, Klaviyo action, landing page polish) hangs off this.

**Honest risk:** this is tight. If any single critical-path item slips by a week, V1 slips proportionally. No buffer. **Check-in at 40% elapsed** (i.e. Week 4 of a 10-week target, Week 5 of 12): if critical path is behind, activate cut-list in sequence before extending the timeline.

**Unmodeled overhead to acknowledge explicitly:**
- Shopify OAuth + app listing review: 1–3 weeks wall time (Shopify-controlled)
- Gorgias OAuth app setup: 3–5 days
- Klaviyo private app auth: 1–2 days
- Slack app review (if public-distributed): 1 week; skippable by shipping workspace-scoped install only at first
- AI leverage does NOT compress these — they're third-party gates. Start these applications Week -2 or earlier so wall-time overlaps with build. Plan assumes this parallelism.

**Cut-list (pre-committed, triggered if Week 4 check-in shows critical path behind):**
1. Drop interactive sandbox → use Loom degrade rule
2. Drop 3rd stubbed template
3. Drop 2nd stubbed template (ship 2 seeded + 1 stubbed)
4. Drop measurement primitive from V1 (ship as alerts-only; measurement V1.1)
5. Kill Klaviyo action binding (ship Gorgias tag + Slack alert only)

Not a buffer — a pre-committed de-scope sequence. Saves decision-cost under pressure.
