  <behavior>
    CONF-01 (relationship creation):
    - confirmSuggestion creates CONNECTED_ENTITIES relationship via entityRelationshipService.addRelationship with linkSource=SourceType.IDENTITY_MATCH and definitionId=null

    CONF-02 (5 cluster cases):
    - Case 1: Neither entity clustered -> creates new IdentityClusterEntity + 2 IdentityClusterMemberEntity rows, memberCount=2
    - Case 2: Source clustered, target not -> adds target as new member, increments surviving cluster memberCount
    - Case 3: Target clustered, source not -> adds source as new member, increments surviving cluster memberCount
    - Case 4: Both in different clusters -> merges smaller into larger (hard-delete dissolving members, re-create in surviving, update memberCount, soft-delete dissolving cluster)
    - Case 5: Both in same cluster -> no cluster mutations (no saves, no deletes)

    CONF-03 (merge):
    - On merge, surviving cluster = higher memberCount (on tie, keep source entity's cluster)
    - Dissolving cluster's members are hard-deleted then re-inserted into surviving cluster
    - Dissolving cluster is soft-deleted (deleted=true, deletedAt set)
    - Surviving cluster memberCount = old count + dissolving members count

    CONF-04 (reject):
    - rejectSuggestion transitions PENDING -> REJECTED with resolvedBy=userId, resolvedAt=now
    - Rejection writes rejectionSignals snapshot and soft-deletes the suggestion
    - Activity logged for rejection

    CONF-05 (state guards):
    - confirmSuggestion on CONFIRMED suggestion -> ConflictException
    - confirmSuggestion on REJECTED suggestion -> ConflictException
    - rejectSuggestion on REJECTED suggestion -> ConflictException
    - rejectSuggestion on CONFIRMED suggestion -> ConflictException

    Activity logging:
    - confirmSuggestion logs activity with Activity.MATCH_SUGGESTION, OperationType.UPDATE, details including "action" to "confirmed", clusterId, clusterMemberCount
    - rejectSuggestion logs activity with Activity.MATCH_SUGGESTION, OperationType.UPDATE, details including "action" to "rejected"

    Notification:
    - confirmSuggestion publishes notification via notificationService.createInternalNotification with userId=null (broadcast), type=REVIEW_REQUEST, referenceType=ENTITY_RESOLUTION, referenceId=suggestionId

    Create `src/main/kotlin/riven/core/service/identity/IdentityConfirmationService.kt`:

    Package: `riven.core.service.identity`
    Annotations: `@Service`

    Constructor injection:
    - matchSuggestionRepository: MatchSuggestionRepository

    Public methods (both with `@Transactional` and `@PreAuthorize("@workspaceSecurity.hasWorkspace(#workspaceId)")`):

    1. `confirmSuggestion(workspaceId: UUID, suggestionId: UUID): MatchSuggestion`
       - `val userId = authTokenService.getUserId()`
       - `val entity = ServiceUtil.findOrThrow { matchSuggestionRepository.findById(suggestionId) }`
       - `validateConfirmable(entity)` — throws ConflictException if status != PENDING
       - `createRelationship(entity)` — calls entityRelationshipService.addRelationship
       - `val cluster = resolveCluster(entity, userId)` — the 5-case cluster resolution
       - Apply confirmation state: status=CONFIRMED, resolvedBy=userId, resolvedAt=now (NOT soft-deleted)
       - `matchSuggestionRepository.save(entity)`
       - Log activity
       - Publish notification with entity pair + cluster member count
       - Return entity.toModel()

    2. `rejectSuggestion(workspaceId: UUID, suggestionId: UUID): MatchSuggestion`
       - `val userId = authTokenService.getUserId()`
       - `val entity = ServiceUtil.findOrThrow { matchSuggestionRepository.findById(suggestionId) }`
       - `validateRejectable(entity)` — throws ConflictException if status != PENDING
       - Apply rejection: write rejectionSignals snapshot, set status=REJECTED, resolvedBy, resolvedAt, soft-delete (deleted=true, deletedAt=now)
       - `matchSuggestionRepository.save(entity)`
       - Log activity
       - Return entity.toModel()

    Private methods (organized with section comments):

    `// ------ State validation ------`
    - `validateConfirmable(entity)` — ConflictException if not PENDING
    - `validateRejectable(entity)` — ConflictException if not PENDING

    `// ------ Relationship creation ------`
    - `createRelationship(entity)` — calls entityRelationshipService.addRelationship with workspaceId=entity.workspaceId, sourceEntityId=entity.sourceEntityId, request=AddRelationshipRequest(targetEntityId=entity.targetEntityId, definitionId=null, linkSource=SourceType.IDENTITY_MATCH)

    `// ------ Cluster management ------`
    - `resolveCluster(entity, userId): IdentityClusterEntity` — implements the 5-case when block:
      - Case 5: both same cluster -> return the cluster, no mutations
      - Case 4: both different clusters -> mergeClusters(), return surviving
      - Case 2: source clustered only -> addMemberToCluster(), return cluster
      - Case 3: target clustered only -> addMemberToCluster(), return cluster
      - Case 1: neither -> createClusterWithMembers(), return new cluster
    - `createClusterWithMembers(workspaceId, sourceEntityId, targetEntityId, entity, userId): IdentityClusterEntity`
      - Create IdentityClusterEntity with name from resolveClusterName(entity), memberCount=2
      - Save cluster, save 2 IdentityClusterMemberEntity rows
      - Return saved cluster
    - `addMemberToCluster(cluster, entityId, userId): IdentityClusterEntity`
      - Save new IdentityClusterMemberEntity, increment cluster.memberCount, save cluster
      - Return updated cluster
    - `mergeClusters(sourceClusterId, targetClusterId, userId): IdentityClusterEntity`
      - Load both clusters via findOrThrow
      - Determine surviving vs dissolving: surviving = higher memberCount, tie goes to source cluster
      - Load dissolving members via findByClusterId
      - Hard-delete dissolving members via deleteByClusterId
      - Re-insert each dissolving member into surviving cluster (new IdentityClusterMemberEntity with surviving clusterId, preserving original joinedAt and joinedBy)
      - Update surviving.memberCount += dissolving members count
      - Soft-delete dissolving cluster (deleted=true, deletedAt=now)
      - Save both clusters
      - Return surviving cluster

    `// ------ Cluster naming ------`
    - `resolveClusterName(entity: MatchSuggestionEntity): String?`
      - Look through entity.signals list for an entry where type == "NAME"
      - If found, return the sourceValue from that signal entry
      - If not found, return null (IdentityClusterEntity.name is nullable)

    `// ------ Activity logging ------`
    - `logConfirmationActivity(entity, userId, cluster)`
    - `logRejectionActivity(entity, userId)`

    `// ------ Notification ------`
    - `publishConfirmationNotification(entity, cluster)` — calls notificationService.createInternalNotification with:
      - workspaceId = entity.workspaceId
      - userId = null (broadcast to all workspace members)
      - type = NotificationType.REVIEW_REQUEST
      - content = NotificationContent.ReviewRequest(title="Identity match confirmed", message="[source entity display] and [target entity display] have been linked. Identity cluster now has N members.", priority=ReviewPriority.NORMAL)
      - referenceType = NotificationReferenceType.ENTITY_RESOLUTION
      - referenceId = entity.id (the suggestion ID)
      - For entity display names in the message, use the NAME signal from entity.signals if available, otherwise "Entity {id.toString().take(8)}"
