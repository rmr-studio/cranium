**Overview**

Community summaries are pre-computed summaries of a cluster of related entities,
- A cluster is formed by a group of **pages**
	-  A cluster will never reference individual **source files** 
	- The clustering mechanism is forced by running over the wikilink graph between pages
		- Leidin 
		- GraphRAG-style
		- Directory Structure
		- CODEOWNERS (ie. Relevant parties who own/associated with a page)

typically within a domain and is the result of an aggregation and computation of atomic pages and source materials, produced by an LLM at index time. It sits at the root index of a particular domain and is the step in-between it and atomic chunks [[5. Atomic Page Synthesis]]
It acts as a first context layer when performing information retrieval through queries or cross linking (ie. Query time) to ensure that the model never needs to go through and manually scan raw chunks of information when answering global questions

This is an abstract concept explored in graph based RAG systems such as
- [GraphRag/LazyGraphRag](https://github.com/microsoft/graphrag)
This is derived from where corpus are generated and indexed about a particular domain to give base level knowledge 
- ie `Wiki/<Domain>/index.md`

# **Token Usage**
This approach during the generation of an LLM wiki will move tokens away from query time capabilities
  - Prior Methods
	  - **Naive vector RAG**: top-k chunks → many irrelevant near-neighbors → LLM re-reasons relationships every query.
  - Community Summaries
	  - one summary already encodes "these 200 chunks are about X, related to Y via Z." LLM reads the synthesis, not the raw evidence. Drill to atomic chunks only when local specificity needed.

This will rapidly increase token usage during index time, given the larger number of resources required to index and provide summaries of related entities

**Use Case**
  1. Global questions ("what are major themes") — no single chunk contains the answer; flat retrieval fundamentally can't.
  2. Multi-hop reasoning — relationships pre-traced in graph, summarized in cluster, so LLM skips the traversal.
  3. Context-window economics — fewer, denser tokens per query.
